# 3000 股全市场排名扩容设计

## 目标

将现有 800 股真实数据大池扩容为 3000 股。所有股票使用相同的 4 年历史、特征、模型和综合评分口径，最终在同一股票池内计算分位和排名。公网访客仍只读取每日预计算快照，不在请求路径中训练模型。

## 范围

- `DataRequest.max_symbols` 默认值改为 3000。
- 本地界面的真实数据上限和默认值改为 3000。
- 公网模式、提示文案、README、快照脚本和 GitHub Actions 统一改为 3000。
- 不改变目标持有期、特征定义、模型组合、回测门槛和最终推荐数量上限。
- 不引入“先粗筛 3000、再深度计算 800”的两套评分口径。

## 股票池选择

数据源首先获取完整 A 股上市公司列表，然后在上证主板、深证主板、创业板和科创板中选取最多 3000 只。继续使用现有板块均衡逻辑；某板块数量不足时，用其他已选板块的股票补足，直到 3000 只或数据源可用股票耗尽。

每份线上快照必须显示请求数量、行情覆盖数量、评分覆盖数量和失败补齐数量。推荐列表只能声称实际进入统一评分的覆盖数量。

## 每日更新与失败补齐

GitHub Actions 在每个交易日北京时间 18:30 运行。任务先下载上一版已验证的全量真实行情作为回退基线，再并发更新当日 3000 只股票。单股当日接口失败时，使用上一版真实行情补齐，并在 manifest 中记录；不使用演示数据或占位数据。

上传新快照前执行严格质量门槛：

1. 行情覆盖必须为 3000 只。
2. 最新评分覆盖必须为 3000 只。
3. 推荐结果必须按综合评分严格降序。
4. 快照文件必须全部通过 SHA-256 校验。

任一门槛失败时，任务失败并保留上一版线上快照，不发布部分股票池。

## 快照结构

快照继续包含小型公网读取文件：推荐列表、最新评分、回测曲线、精简价格历史和 manifest。另外增加构建专用的全量真实行情文件，仅供每日 GitHub Actions 补齐失败股票；Hugging Face Space 不下载该大文件，避免增加访客启动时间和内存。

manifest 升级后区分“应用必需文件”和“构建回退文件”。公网加载器只校验和下载应用必需文件，构建脚本在自动更新时校验全部文件。

## 性能与资源

- AKShare 继续使用有界并发、单股重试和 Tencent 日线回退。
- GitHub Actions 任务超时从 330 分钟调整为 360 分钟。
- 特征和模型仍在单个任务中统一计算，不分割成不同评分口径。
- 生成过程使用精简列、Parquet 压缩和必要的数值类型优化，将峰值内存控制在免费 Runner 可用范围内。
- 公网应用仍延迟加载快照，推荐页不下载全量构建回退文件。

## 验证

- 单元测试覆盖 3000 默认值、板块补齐、快照覆盖门槛、上一版真实行情回退和公网按需下载。
- 执行完整 Python 编译和测试集。
- 本地生成一份 3000 股快照，确认行情和评分均为 3000 只。
- 发布 Hugging Face Dataset 后，确认 manifest 和校验值可公开读取。
- 重新构建 Hugging Face Space，验证健康检查、推荐列表、实际覆盖数和个股评估。
- 手动触发一次 GitHub Actions 3000 股更新，成功后才认为线上扩容完成。

## 已知限制

- 3000 股全量更新明显比 800 股慢，但该开销只发生在每日后台任务，不会转移到访客请求。
- 免费 Runner 和数据源频率均无 SLA；任务超时或数据质量不足时，系统保留上一版合格快照。
- 推荐覆盖扩展不代表模型胜率必然提高；回测门槛和风险提示继续生效。
